# ADR-002: bd as the Bus — Plain Notes Ledger and Shared Design Field

## Status
accepted

## Context
The post-rework /intuition audit (bd-cnp) flagged two state-shape tensions in
how the pipeline uses bd:

- T10: a task's design field is a shared mutable record with four writer
  roles (brainstorming creation, writing-plans repair, SRE-recommended edits,
  lead auto-promotion) and three reader roles, including dispatch-time
  parsing of the `Executor: opus` control line inside prose. No schema
  validation exists between writers and readers.
- T11: epic bd notes carry three record types (GATE STATE blocks,
  PLAN-IMPACT notices, promotion notes) in one append-only prose channel,
  distinguished only by prefix; the latest gate is found by position.

bd's data model is plain fields plus notes. The prose ledger is also what
makes a lane readable at a glance by the rotating governor — the primary
consumer (Mode 1/2 operating model; principles: artifacts over conversation
state, human-readable evidence).

## Decision
We will keep bd's plain design field and notes channel as the pipeline's bus.
Writer safety comes from protocol, not schema: writers act sequentially
(plan → repair → promote), the promotion-line convention stays single-sourced
in pipeline-constants.md, and record types keep their fixed prose prefixes
(`GATE STATE (`, `PLAN-IMPACT:`) per loop-interfaces.md. Resume points read
the latest GATE STATE block (added in bd-cnp R2).

Re-examination trigger: an observed reconstruction failure (fresh session
misreads a lane's state) or observed writer interleaving (a promotion or
amendment landing mid-dispatch), either of which reopens the
structured-channel question.

## Consequences
- Zero new machinery; lanes stay governor-readable as plain prose.
- The parsed control line continues to live inside prose — accepted risk,
  bounded by the single-sourced convention and sequential-writer protocol.
- Record identity stays convention-based; tooling that wants to query gates
  must scan prefixes.
