# ADR-003: Executor Guard Cluster Is the Interface Specification

## Status
accepted

## Context
The post-rework /intuition audit (bd-cnp T14) examined the six guard
mechanisms concentrated on the executor interface: the one-liner return
contract with negative-space rules, first-word-only parsing, the
`Executor: opus` auto-promotion rung, consecutive-BLOCKED thresholds, the
mandatory two-stage review after every DONE, and the executor's own output
gate. The audit's Pass 9 probe found no mechanism bypass: the executor uses
the same dispatch transport as every other agent and is unique only in that
its return is parsed for control flow.

The guarded failure mode — runtime format drift and requirement watering
under blocker pressure — is external model behavior, empirically confirmed
live (bd-0he: an executor emitted SRE-style prose in place of its return
contract during the very epic fixing that bug), and recurred across three
successive execution-model reworks (teammate, subagent, hybrid).

## Decision
We will treat the guard cluster as the specification of an interface to an
unreliable component, not as a workaround pile to be cleaned up. Guards are
not pruned on inspection or inactivity. Removal of any guard requires the
writing-skills A/B re-baselining method (resources/testing-methodology.md)
run against live executor dispatches: a baseline without the guard must show
the failure no longer reproduces at the executor's model tier.

## Consequences
- Silent executor drift stays converted into visible escalation, at the cost
  of ~6 mechanisms' maintenance on one interface.
- agents/executor.md remains a hot file relative to other agents; that heat
  is accepted as the price of the only parsed-for-control-flow interface.
- Future executor-model upgrades get a defined, evidence-based path to
  simplify the interface (the A/B campaign), rather than ad-hoc pruning.
